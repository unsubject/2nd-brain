## Loose ends + notes

- **Worker outer-try catch clobbers `processed` rows on downstream failure.** `src/worker.ts:tick()` extends its outer try through `generateLinks` *after* `saveProcessingResult` has already set `status='processed'`. If `generateLinks` throws, the outer catch fires `markProcessingError` which flips status back to `'error'` and (post-#53) writes a link-generation error string into `last_error` attached to a row whose primary processing actually succeeded. Pre-existing bug, made louder by PR #53. New Build task tracks it. Two fixes: (a) move `generateLinks` outside the outer try, or (b) guard `markProcessingError` with `WHERE processing_status != 'processed'`.

- **No `resetErroredEntries` helper.** Archive has `resetErroredArtifacts()` for one-call re-queue of errored rows. Journal has nothing — re-queue today is manual SQL. Tracked on the post-merge verify Build task as a "consider adding."

- **MCP `2nd-brain-pg` transaction wedge fired twice this session** — once inherited from yesterday, once self-inflicted from a `n_entries` column-name typo (actual column is `entry_count`). The MCP's read-only transactions don't auto-rollback on error and `execute_rollback` needs a `transaction_id` that isn't returned on failure. Forces a full `/mcp` reconnect every time. Workflow tax until the MCP server is fixed; the cheap defense is `describe_table` before any exploratory query on a table whose schema isn't fresh in context.
