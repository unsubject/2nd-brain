## How we found it

The 2026-05-12 handoff expected a clean drain "~1 hour after handoff." 3 days later there were 32 `error` rows and 1 stuck `processing`, all Chinese-language. Data-state triage was decisive:

- **Error rows**: `chunks=4-8`, `has_embedding=true`, `entities=0` → failed *after* chunking + embedding, *before* entity-extraction writes. That's worker.ts step 7.
- **Stuck row**: `chunks=0`, `has_embedding=false`, `entities=0` → died at step 1 (or never got past the `status='processing'` flip). Different bug class — picked up cleanly on retry.
- **Spot-check on a processed Chinese row** (`f42bb9f0`): 4 chunks, **39 entities**, summary embedded, structured English summary matching `videos.db` shape → confirmed `sourceSummary` plumbing works *and* the entity extractor handles Chinese in general.

So the failure wasn't categorical "Chinese fails" — it was 9% of Chinese rows hitting some specific wall. Railway log archaeology for the 47-second failure window only surfaced a flood of unrelated `PayloadTooLargeError`s and one clean successful trace. `console.error` lines from the actual failures were either rotated or filter-suppressed.

**Pivot: live reproduction.** Reset one error row (`316e6eda`, shortest `raw_source` in the error set) to `pending`. Worker picked it up in ~60s and produced:

```
[archive] entities tokens: in=4970 out=2048
[archive] Error processing "中山美穗…": SyntaxError: Unterminated string in JSON at position 5638
    at JSON.parse (<anonymous>)
    at extractEntities (/app/dist/archive/processor.js:130:25)
```

`out=2048` hit the cap exactly → response truncated mid-string → `JSON.parse` threw `SyntaxError` → worker catch wrote `processing_status='error'`. Same row on retry after deploy: `out=3055 finish=stop` → 93 entities. Fix validated.

## Loose ends + notes

- **`PayloadTooLargeError` flood** is still un-investigated. Body-parser defaults to 100kb on every route except `/archive/import/youtube` (PR #40 set the 10mb override). Suspect `/archive/import/notion-api` getting hit with >100kb essay payloads — matches the unexplained +283 Notion rows the 2026-05-12 handoff flagged. New Build task tracks this.

- **`public_artifact` has no error-message column.** When the worker catches, the exception text is discarded; the only forensic trail is Railway logs, which roll. Adding `last_error TEXT` is a one-migration change that would make this kind of triage a SQL query instead of a log archaeology session. New Build task tracks it.

- **MCP postgres connection wedge.** First query in the session referenced a non-existent column. The MCP wraps every query in `BEGIN TRANSACTION READ ONLY` and doesn't `ROLLBACK` on in-transaction errors, so the next query (and every one after) hits `current transaction is aborted, commands ignored`. The MCP's own `execute_rollback` needs a `transaction_id` it doesn't return on the failed query — no recovery from the client side, the user had to `/mcp` reconnect. Bug in the `2nd-brain-pg` MCP server, not in our code.
